# Plan: Fix Nogood ID Collisions on Deletion

**Issue:** #26 — Nogood IDs assume append-only list, deletion causes collisions

## Requirements Analysis

**The bug:** `network.py:258` generates nogood IDs as `f"nogood-{len(self.nogoods) + 1:03d}"`. If a nogood is ever removed from the list, the next `add_nogood` call reuses an existing ID, corrupting the contradiction history.

**Why it matters:** Although no deletion code exists today, the ID generation is silently fragile. Anyone adding nogood management later (resolution cleanup, GC of resolved contradictions) would introduce data corruption with no warning. The defect also blocks 3 derived beliefs in the knowledge base.

**What we're building:** Replace the length-based ID derivation with a monotonically increasing counter on the `Network` object, derived from the max existing nogood number on load. No new persistence mechanism needed — the counter is always recoverable from the data.

**What we're NOT building:** Nogood deletion itself. This fix makes the ID scheme *safe* for future deletion; it doesn't add deletion.

## Key Design Decisions

### 1. Monotonic counter vs. UUIDs

**Decision: Monotonic counter.** UUIDs would also solve the uniqueness problem but would break the human-readable `nogood-001` format that appears in logs, audit trails, and exported JSON. The counter preserves readability with minimal complexity.

### 2. Persist the counter vs. derive on load

**Decision: Derive on load.** Rather than adding a metadata table to SQLite or a field to JSON exports, compute `_next_nogood_id` by scanning existing nogoods for the max numeric suffix. This is simpler (no schema migration), always consistent with actual data, and the nogood list is small enough that O(n) on load is negligible.

### 3. Where to put the derivation logic

**Decision: Private helper method on Network.** A `_compute_next_nogood_id()` method avoids duplicating the regex/parsing logic across `storage.py` and `api.py`. Callers (storage load, JSON import) call it after populating `self.nogoods`.

### 4. Handling prefixed IDs from agent imports

The `import_agent.py:273` path creates IDs like `agent_name:nogood-001`. These are prefixed and don't conflict with the unprefixed `nogood-NNN` pattern. The counter only governs unprefixed IDs generated by `add_nogood`. **No change needed to import_agent.py.**

**Helper method spec:**
```python
import re

def _compute_next_nogood_id(self) -> None:
    """Set _next_nogood_id from existing nogoods (safe after deletions)."""
    max_id = 0
    for ng in self.nogoods:
        m = re.fullmatch(r"nogood-(\d+)", ng.id)
        if m:
            max_id = max(max_id, int(m.group(1)))
    self._next_nogood_id = max_id + 1
```

Using `re.fullmatch` ensures prefixed IDs like `agent:nogood-001` are excluded (they don't match the full pattern).

### Verification: Bidirectional consistency

- **Write path (add_nogood):** Uses `self._next_nogood_id`, increments it. Always produces unique unprefixed IDs regardless of list mutations.
- **Read path (storage load):** After loading all nogoods from SQLite, calls `_compute_next_nogood_id()` which scans the loaded IDs. Any subsequent `add_nogood` picks up where the data left off.
- **Import path (JSON):** Same as read path — `_compute_next_nogood_id()` after bulk import.
- **Agent import path:** Uses prefixed IDs, bypasses `add_nogood`, doesn't touch the counter. The `fullmatch` regex excludes these. No conflict.
- **Existing test (`test_network.py:243`):** Still passes — first nogood in a fresh network is still `nogood-001`.

## SELF-REVIEW

### What went well
- The codebase is well-structured and small enough to fully trace the nogood lifecycle (creation → persistence → load → export → import) in a single pass.
- The "derive on load" approach avoids schema changes, making this a minimal-surface-area fix.

### What information was missing
- No clarity on whether there are plans to add nogood deletion. The fix is defensive regardless, but knowing the timeline would help prioritize.
- No performance benchmarks on nogood list sizes. The O(n) scan is assumed negligible but unverified.

### What would help next time
- A test that exercises the full save/load/add cycle would have been useful to reference. The existing tests test `add_nogood` in isolation without persistence round-tripping.
